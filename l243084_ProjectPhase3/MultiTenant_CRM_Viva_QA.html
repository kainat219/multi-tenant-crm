Architecture answer

Frontend pages call Express API routes using fetch. Express receives JSON, validates inputs, uses parameterized SQL Server queries, and returns JSON back to the page. The database stores all permanent data and relationships.

Database Design and Relationships

Main parent tables

  • Tenants is the parent for company data.
  • Users belongs to one tenant and has a role.
  • Subscription_Plans is linked to tenants and billing.
  • Pipeline_Stages belongs to a tenant and is used by opportunities.

CRM working tables

  • Leads, Contacts, Accounts, Opportunities store sales data.
  • Activities track tasks, meetings, and calls.
  • Support_Tickets and Ticket_Comments handle support.
  • Audit_Logs and Notifications support monitoring.

Constraints answer

The database should use primary keys for every table, foreign keys for relationships, NOT NULL for required fields, UNIQUE for values like username, email, tenant domain, or invoice number, DEFAULT for dates like created_at, and CHECK constraints for valid statuses, roles, amount values, and probability percentages.

Normalization up to BCNF

QHow did you normalize the database?
AI separated each real-world thing into its own table: tenants, users, leads, accounts, contacts, opportunities, plans, billing, tickets, and so on. This avoids repeating the same data again and again.
QGive a 3NF or BCNF example from your project.
AIn Opportunities I store stage_id, not stage_name and probability directly. Stage details are in Pipeline_Stages. So stage_id determines the stage details, and no non-key field depends on another non-key field.

SQL Queries Your Teacher May Ask

QWhere are CRUD operations used?
ACRUD is used in almost every module. For example, Leads has add lead, get leads, update lead, and delete lead. Contacts, accounts, users, tickets, plans, opportunities, and billing also have CRUD routes.

Create and read example

INSERT INTO Leads
  (tenant_id, first_name, last_name, email, phone,
   company, source, status, notes, assigned_user_id)
VALUES
  (@tenant_id, @first_name, @last_name, @email,
   @phone, @company, @source, @status, @notes,
   @assigned_user_id);

SELECT l.*, u.first_name + ' ' + u.last_name AS assigned_user
FROM Leads l
LEFT JOIN Users u ON l.assigned_user_id = u.user_id
WHERE l.tenant_id = @tenant_id;

Update and delete example

UPDATE Opportunities
SET status = @status,
    last_updated = GETDATE()
WHERE opportunity_id = @opportunity_id;

DELETE FROM Support_Tickets
WHERE ticket_id = @ticket_id;
QWhich joins are used in your project?
AI use JOIN and LEFT JOIN to show readable names instead of only IDs. For example, opportunities join tenants, accounts, contacts, users, and pipeline stages so the dashboard can show company name, contact name, assigned user, and stage name.
SELECT
  o.opportunity_name,
  o.deal_value,
  t.company_name,
  a.account_name,
  c.first_name + ' ' + c.last_name AS contact_name,
  u.first_name + ' ' + u.last_name AS assigned_user,
  ps.stage_name
FROM Opportunities o
LEFT JOIN Tenants t ON o.tenant_id = t.tenant_id
LEFT JOIN Accounts a ON o.account_id = a.account_id
LEFT JOIN Contacts c ON o.contact_id = c.contact_id
LEFT JOIN Users u ON o.assigned_user_id = u.user_id
LEFT JOIN Pipeline_Stages ps ON o.stage_id = ps.stage_id;
QWhere did you use grouping?
AThe dashboard uses GROUP BY for reports. Revenue is grouped by tenant company, and pipeline value is grouped by tenant company. This gives summary values, not only raw records.
SELECT t.company_name,
       SUM(b.amount) AS total_revenue,
       COUNT(b.billing_id) AS total_invoices
FROM Billing_Records b
JOIN Tenants t ON b.tenant_id = t.tenant_id
WHERE b.payment_status = 'paid'
GROUP BY t.company_name
ORDER BY total_revenue DESC;

Triggers and Transactions

Important demo note

Your route files show the audit module and the lead conversion workflow. If your SQL Server has manually created triggers or transactions, show them from SQL Server during viva. If the teacher asks for the concept, use the answers below.

QWhat trigger is used in your project?
AThe trigger concept is used for audit logging. When an important table such as Leads, Users, Opportunities, or Billing_Records changes, a database trigger can automatically insert a record into Audit_Logs with action type, table name, tenant_id, user_id, and time. This gives a history without writing audit code on every page.
QWhy use a trigger instead of normal code?
AA trigger runs automatically inside the database. Even if the change comes from another route or admin tool, the audit record is still created. This protects consistency.
CREATE TRIGGER trg_Leads_Audit
ON Leads
AFTER INSERT, UPDATE, DELETE
AS
BEGIN
  INSERT INTO Audit_Logs (tenant_id, action_type, table_name, performed_at)
  SELECT tenant_id, 'LEAD_CHANGED', 'Leads', GETDATE()
  FROM inserted;
END;
QWhere is transaction needed in your project?
ALead conversion is the best transaction example. It inserts a row into Lead_Conversions and also updates the original lead status to converted. Both steps must succeed together. If the insert succeeds but the update fails, the database would become inconsistent, so we rollback.
BEGIN TRANSACTION;

INSERT INTO Lead_Conversions
  (tenant_id, lead_id, contact_id, account_id,
   opportunity_id, converted_by, notes)
VALUES
  (@tenant_id, @lead_id, @contact_id, @account_id,
   @opportunity_id, @converted_by, @notes);

UPDATE Leads
SET status = 'converted',
    last_updated = GETDATE()
WHERE lead_id = @lead_id;

COMMIT;
-- If any step fails: ROLLBACK;
QGive another transaction example.
ARegistration can also be treated as a transaction because the system creates a tenant first and then creates a tenant admin user. If the user creation fails, the tenant creation should also rollback so no empty company is left.

Likely Viva Questions with Simple Answers

QWhat problem does your CRM solve?
AIt helps companies manage customers from first lead to sales opportunity, support ticket, billing, and follow-up activities in one system.
QWhat does multi-tenant mean?
AMany companies use the same application and database, but every company has its own tenant_id. Queries filter by tenant_id so one company does not see another company's data.
QHow does login work?
AThe login route checks username and password in Users, joins Tenants to get company name, checks is_active, updates last_login, and sends the user role back to the frontend.
QHow are roles handled?
AThe user role decides the dashboard. super_admin sees platform-wide pages, tenant_admin sees one tenant's data, and sales_agent sees assigned records.
QHow do you prevent SQL injection?
AThe backend uses parameterized inputs like .input('tenant_id', sql.Int, tenant_id) instead of directly joining user text into SQL. This is safer.
QWhy did you separate Subscription_Plans from Tenants?
ABecause plan details like max_users, max_contacts, price, and features should be stored once. Tenants only store subscription_plan_id. This reduces duplication and supports normalization.
QWhy did you create Pipeline_Stages?
APipeline stages are reusable steps like new, qualified, proposal, won, or lost. Opportunities only store stage_id, so stage names and probabilities are managed in one table.
QWhat is Lead_Conversions used for?
AIt keeps the history of converting a lead into a contact, account, or opportunity. It also records who converted it and when.
QWhat happens when a ticket is resolved?
AThe ticket status is updated and resolved_at is set. That records both the current state and the time of resolution.
QWhere is GETDATE used?
AGETDATE is used for time tracking, such as last_login, last_updated, notification read_at, audit performed_at, and resolved_at.

More Questions They Can Ask

QHow does the dashboard work?
AIt calls backend routes that count leads, contacts, tickets, and tenants. It also calculates revenue and pipeline value using SUM, COUNT, AVG, GROUP BY, and JOIN.
QWhy use LEFT JOIN?
ALEFT JOIN still shows the main record even if optional related data is missing, such as a lead without an assigned user.
QWhat is the primary key?
AIt uniquely identifies each row. Examples are tenant_id, user_id, lead_id, contact_id, account_id, opportunity_id, and ticket_id.
QWhat is a foreign key?
AIt connects two tables. For example, Leads.tenant_id connects a lead to Tenants, and Opportunities.stage_id connects a deal to Pipeline_Stages.
QWhy have Custom_Fields?
AEvery tenant may need extra fields. Instead of changing table columns each time, Custom_Fields stores tenant-specific field definitions.
QWhy have Email_Integrations?
AIt stores each user's email provider and address, so the CRM can later support email sync and customer communication tracking.
QWhat is audit logging?
AAudit logging means keeping a record of important actions, such as who changed data, what table changed, and when it happened.
QWhat improvement would you add later?
AI would add password hashing, JWT/session security, stricter database constraints, full transaction wrapping for multi-step operations, and more reports.

Safe closing answer

The main database idea in my project is that tenant_id separates company data, foreign keys connect related CRM modules, joins create readable screens, grouping creates reports, and transactions protect multi-step work like lead conversion.
